# Дизайн: аудит целостности кода, надёжность и скорость разработки

**Дата:** 2026-04-12  
**Статус:** черновик спецификации (brainstorming)  
**Приоритеты продукта владельца:** **D** — надёжность и сопровождение (меньше мёртвого/устаревшего кода, ясные контракты, тесты); **B** — скорость и удобство разработки (понятные границы модулей, меньше нагромождений).

**Каноничная архитектура продукта:** FastAPI (`backend/`) + React SPA (`frontend/`), бизнес-логика в `core/`, интеграции в `services/`, данные в `db/`. Документация: `README.md`, `docs/current/`, встроенная справка `frontend/src/docs/user/`. Материалы в `docs/archive/` не являются источником истины о текущем продукте.

---

## 1. Цели и критерии успеха

### 1.1 Цели

1. Сделать репозиторий **целостнее**: явные слои и зависимости, меньше «протекающих» обязанностей между `backend/api`, `core`, `services`.
2. Снизить **накопленный шум**: устаревшие комментарии/пути, неиспользуемые ветки кода, расхождение док ↔ код.
3. Усилить **D**: предсказуемое поведение при ошибках, тесты на критичных путях, меньше неявных зависимостей.
4. Усилить **B**: новый участник или ты через полгода быстрее находит «куда править» и не ломает соседние модули.

### 1.2 Измеримые ориентиры (не жёсткие KPI на старте)

- Есть **единая политика слоёв** (кто может импортировать кого) и она проверяется хотя бы ручным чеклистом в PR; по желанию — линтер/скрипт.
- **Инвентаризация наследия**: список «архив / к удалению / к переносу в docs» согласован и не противоречит README.
- **Критичные потоки** (минимум: аутентификация/сессия, генерация промпта, выбор техник, сохранение в библиотеку) имеют **явные тесты или контрактные проверки** там, где сейчас пробелы.
- Докстринги и заголовки модулей **не ссылаются на Streamlit как на равноправный UI**, если Streamlit не входит в целевой контур разработки.

### 1.3 Вне области этого дизайна

- Новые пользовательские фичи, не следующие напрямую из устранения техдолга (их можно добавить в `docs/current/IMPROVEMENT_PLAN.md` отдельно).
- Полная перепись архитектуры или смена стека.

## 2. Три подхода и рекомендация

### Вариант A — Только инкрементальный бэклог

Небольшие PR без явной «карты» зависимостей: по мере работы чистим файлы и доки.

- **Плюсы:** низкий риск, не отвлекает от фич.  
- **Минусы:** мало **B**: границы модулей не проясняются; **D** улучшается медленно и неравномерно.

### Вариант B — «Замещение» с явными границами (strangler)

Сначала фиксируются правила слоёв и список модулей-«горячих точек», затем изменения только через эти правила (перенос логики из роутеров в `core`, выравнивание `services`).

- **Плюсы:** сильный эффект на **B** и **D**, накопление регрессий контролируется тестами.  
- **Минусы:** нужна дисциплина и время на первую фазу описания границ.

### Вариант C — Большой взрыв (массовое удаление и рефакторинг за один проход)

- **Плюсы:** быстро убрать видимый мусор.  
- **Минусы:** высокий риск поломок, плохая согласуемость с историей git и ревью; **не рекомендуется**.

### Рекомендация

**B с элементами A:** один короткий **фазовый план** (границы + инвентаризация наследия + тестовые защиты), дальше **инкрементальные PR** по приоритету. Это лучше балансирует **D** и **B**, чем чистый A или C.

## 3. Текущий контекст (наблюдения)

- `backend/main.py` монтирует API на `/api` и отдаёт SPA с `frontend/dist` — структура ясная.
- В репозитории **не обнаружен** каталог `app/` с исходниками Streamlit (в отличие от упоминаний в README и ряде комментариев); при аудите нужно **зафиксировать фактическое состояние** (удалён / в другой ветке / только у части клонов) и **синхронизировать документацию**.
- Примеры устаревших формулировок относительно актуального контура: `config/__init__.py` («app (Streamlit) and backend»), комментарии в `services/llm_client.py` про синхронный клиент для Streamlit, `core/parsing.py` про разделение с архивным UI.
- Тесты сосредоточены в `tests/` (~10 модулей); покрытие не претендует на полноту всех API-роутеров — для **D** имеет смысл целенаправленно расширить критичные зоны.
- Существующий бэклог `docs/current/IMPROVEMENT_PLAN.md` дополняет этот дизайн продуктовыми пунктами (Q&A flow, метрики, 401 и т.д.) и **не заменяет** технический аудит слоёв.

## 4. Архитектура процесса аудита (не целевой runtime)

Процесс разбит на **изолированные шаги**, каждый даёт артефакт для ревью:

```text
[Discovery] → [Classify & policy] → [Guardrails + tests] → [Execute refactors] → [Docs sync]
```

- **Discovery:** статический обзор импортов, дубликатов, «сиротских» файлов, расхождений README/docs ↔ дерево репозитория.
- **Classify & policy:** таблица: *оставить / переписать док / удалить / перенести в archive*. Отдельно — **политика слоёв** (например: `backend/api` → тонкие адаптеры; `core` → чистая логика без HTTP; `services` → внешние системы и общие клиенты).
- **Guardrails:** минимальные тесты/контракты перед выноской больших кусков.
- **Execute:** серия небольших PR.
- **Docs sync:** один проход по `README`, `docs/current/`, пользовательской справке при изменении границ или удалении наследия.

## 5. Компоненты и границы (целевое понимание)

| Единица | Назначение | Зависимости (целевые) |
|--------|------------|------------------------|
| `backend/api/*` | HTTP, схемы запросов/ответов, вызов сервисов/`core` | `core`, `services`, `db`, `config` |
| `core/*` | Доменная логика промптов, техник, спецификаций, парсинга | минимум внешних SDK; без FastAPI-типов в идеале |
| `services/*` | LLM-клиент, каталоги, оркестрация вызовов к внешним API | `core`, `config`, сторонние пакеты |
| `db/*` | персистентность | `config`, SQL/драйверы |
| `frontend/src/*` | UI, клиент API | контракт REST из `frontend/src/api` |

Для каждого крупного файла в ходе аудита задаются вопросы: *одна ответственность? понятен ли публичный API модуля? можно ли протестировать без поднятия всего приложения?*

## 6. Потоки данных и ошибки

Аудит **не меняет** пользовательские потоки данных сам по себе. Рефакторинги обязаны сохранять:

- контракты REST (статусы, тела ошибок);
- поведение генерации и сохранения в библиотеку.

Для **D** в спеке фиксируется требование: при изменении обработки ошибок в API — **явно** задокументировать в PR и при необходимости добавить тест на код ответа (см. также пункт про 401 в `IMPROVEMENT_PLAN.md`).

---

## 7. Тестирование

1. **Сохранить и расширить** существующие юнит-тесты `core` (парсинг, техники, метрики качества и т.д.).
2. **Добавить** тонкие интеграционные или контрактные тесты для 2–3 критичных эндпоинтов (по согласованию в плане реализации), если сейчас пробелы мешают рефакторингу роутеров.
3. **Регрессия:** прогон существующего набора `pytest` обязателен перед мержем каждого этапа.

Фронтенд: при затрагивании `frontend/src/api/client.ts` или типов — минимум ручной чеклист или существующие проверки сборки (`npm run build`), если в проекте приняты.

---

## 8. Риски и смягчение

| Риск | Смягчение |
|------|-----------|
| Удаление «мёртвого» кода, которое вызывается скриптами вне CI | Поиск по репо + grep вызовов + документация скриптов в `scripts/` |
| Расхождение Streamlit в доках и в дереве | Явная инвентаризация в первом PR аудита |
| Перенос логики ломает стриминг/таймауты LLM | Тесты и поэтапные PR; не смешивать рефакторинг с изменением промптов |
