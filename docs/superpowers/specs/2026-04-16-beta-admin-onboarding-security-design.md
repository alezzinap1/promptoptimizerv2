# Дизайн: закрытая бета — админка, онбординг, маршруты «главной», безопасность

**Дата:** 2026-04-16  
**Статус:** согласовано в диалоге (разделы 1–5 одобрены)  
**Продукт:** Prompt Optimizer (FastAPI + React SPA, см. корневой `README.md`).  
**Контекст запуска:** **закрытая бета**, один инстанс, **не** публичный SaaS «для всех»; регистрация **открыта по URL** (URL не афишируется).

---

## Резюме решений

| Тема | Решение |
|------|---------|
| Горизонт запуска | Закрытая бета / controlled beta (`docs/current/PRODUCTION_CHECKLIST.md` остаётся ориентиром по зрелости). |
| Админы | Флаг **`is_admin`** в БД; первый админ — одноразовая процедура (скрипт/SQL/миграция). |
| Админка MVP | **A+B+C:** список с поиском, карточка пользователя с лимитами/trial и действиями, агрегаты событий **без** текста промптов и без сырого чата. |
| Онбординг | Публичная витрина (дружелюбный копирайт), после регистрации — короткий блок предпочтений с «Пропустить», на Home — подсказки первые **N** визитов. |
| «Две главные» | Развести URL: витрина на **`/welcome`**, **`/`** редиректит гостя на `/welcome`, авторизованного на **`/home`**; единственная «рабочая главная» — `/home`. |
| Подход к внедрению | Вертикальные срезы по ценности (сначала читаемая админка и действия, затем события, затем онбординг и копирайт, параллельно — точечное усиление безопасности). |

## 1. Цели, границы, критерии успеха

### 1.1 Цели

- Дать **оператору беты** за короткое время найти пользователя, понять **лимиты/активность/ошибки**, принять решение **бан или сброс лимита** без ручного копания в БД по умолчанию.
- Дать **новому приглашённому пользователю** понятный **первый контакт** (витрина без технической «нудятины») и **мягкий путь** в основной сценарий (предпочтения + подсказки на Home).
- Устранить критическую путаницу **«две домашние страницы»** за счёт **явного разделения** витрины и приложения по маршрутам и редиректам.

### 1.2 Критерии успеха

- **Инцидент ≤ 5 минут:** от входа в админку до действия по пользователю (найти → понять состояние → бан/сброс при необходимости).
- **Первый визит:** пользователь понимает, **что это за продукт** и **что нажать дальше** (демо / вход / регистрация) без чтения README.
- **Первые сессии:** пользователь находит **ключ API** и **где смотреть лимиты**, если это встроено в подсказки или короткий онбординг.

### 1.3 Вне области

- Публичный SaaS, биллинг, сложная RBAC, просмотр **содержимого** сохранённых промптов и чатов из админки.
- Полный compliance-цикл и enterprise zero-trust (отдельный бэклог при необходимости).

## 2. Пользовательские потоки и типовые пробки (контекст)

### 2.1 Анонимный вход

Витрина → демо или регистрация/вход → приложение на `/home`.

**Риски без дизайна:** неясность **демо vs аккаунт**, непонимание роли **API-ключа**. Снимаются копирайтом на витрине и онбордингом на Home.

### 2.2 Первый день с аккаунтом

Регистрация → предпочтения (пропуск) → Home с подсказками.

**Риски:** перегруз интерфейса Home; пользователь не находит **лимиты** и **справку**. Снимаются короткими подсказками и ссылкой на `/help` без дублирования длинного контента.

### 2.3 Оператор беты

Список пользователей → карточка → агрегаты событий → действие.

**Риски сейчас:** отсутствие единого UI под A+B+C. Цель спецификации — закрыть это инкрементально.

## 3. Информационная архитектура и маршруты (устранение «двух главных»)

- **`/welcome`** — публичная витрина: продуктовое описание, CTA, блок про ключ в бытовой формулировке, честное описание ограничений демо (если применимо).
- **`/`** — диспетчер: неавторизованный → **`/welcome`**; авторизованный → **`/home`**.
- **`/home`** — единственная «главная работа»; навигация использует **одно** имя пункта (например «Главная» или «Работа» — выбор в реализации, но без синонимов в разных местах).
- После входа **нигде не вести** пользователя обратно на **`/`** как на рабочую страницу.

---

## 4. Админ-модуль: доступ, API, UI, данные

### 4.1 Доступ

- Только пользователи с **`is_admin = true`**.
- API под префиксом **`/api/admin/`**; на каждом обработчике — проверка сессии и флага. Ответ при отсутствии прав: **403** (или **404** для скрытия факта наличия админки — решение в реализации; зафиксировать один стиль на весь префикс).

### 4.2 Аудит

- Логировать **админские действия**: кто, когда, что сделал, над каким `user_id`. Не писать в аудит **тексты промптов** и **PII** сверх необходимого для расследования.

### 4.3 Экраны SPA

- Маршрут **`/admin`** (или согласованный префикс): недоступен без прав; иначе редирект на `/home` или страница «нет доступа».
- **Список:** пагинация, поиск по **email / id**, сортировка по активности/дате регистрации.
- **Карточка:** регистрация, последняя активность, trial/лимиты в терминах **текущей** продуктовой модели; действия **блок/разблок**, **сброс дневного лимита** (или эквивалент, согласованный с существующей реализацией лимитов).
- **События:** только **агрегаты и типизированные события** (время, тип, коды ошибок, счётчики). **Запрещено:** отображение и хранение в этой ленте **тела промпта**, **completion**, пользовательских вложений как контента.

### 4.4 Источник событий

- По возможности **переиспользовать** существующую продуктовую метрику/события.
- Если не хватает типов — добавить **узкий** append-only журнал (тип, `user_id`, timestamp, безопасный `meta`), не блокируясь на «идеальной» observability-платформе.

## 5. Онбординг и копирайт

### 5.1 Витрина `/welcome`

- Тон: **польза и сценарии**, без стека и внутренних названий модулей.
- Явно: **что даёт демо**, **что даёт аккаунт**, **нужен ли ключ** и что будет, если ключа нет (в продуктовых, не инженерных формулировках).

### 5.2 После регистрации

- Один экран: **3–5 вопросов** предпочтений, **«Пропустить»**, повторная настройка позже в существующих настройках/профиле.

### 5.3 Home, первые N визитов

- Короткие подсказки по цепочке «задача → генерация → библиотека» и отдельно **ключ** и **лимиты**.
- **N** и условие остановки (до кнопки «не показывать») — константы в реализации, подбираемые по обратной связи беты.

### 5.4 Связь со `/help`

- Онбординг **отсылает** в справку, а не копирует длинные разделы.

---

## 6. Безопасность (минимум для беты)

- Защита **админ API** и **аудит** (см. §4).
- **События:** не тащить в админ-ленту **контент пользователя** и лишний PII.
- **Прод:** HTTPS, секреты вне репозитория; не ослаблять **rate limiting** на auth и тяжёлых маршрутах; при необходимости отдельные лимиты на `/api/admin/*`.
- **Бэклог усиления:** CAPTCHA на регистрацию, 2FA для админов, IP allowlist на reverse proxy для `/admin`.

---

## 7. Стратегия внедрения (напоминание)

Рекомендуется **вертикальный срез**: сначала админка (чтение → действия → события), затем маршруты `/welcome` и редиректы с `/`, затем экран предпочтений и подсказки Home, параллельно — пункты из §6 по мере необходимости.

**Следующий шаг процесса:** по правилам brainstorming после утверждения этого файла владельцем — навык **writing-plans** для детального плана реализации (отдельная сессия или сообщение по запросу).
