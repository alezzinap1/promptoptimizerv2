# Дизайн: мета-анализ прогонов, доказательная база и сквозные тренды (C1 → C2)

- Дата: 2026-04-26
- Статус: черновик на ревью владельца продукта
- Связь: продолжает `2026-04-25-eval-stability-mvp-design.md`; опирается на существующие `eval_runs`, `eval_results`, синтез `synthesis.py` и исполнитель `run_executor.py`.

## 1. Зачем эта спека

Текущий продукт уже:

- прогоняет промпт(ы) N раз, судит по рубрике, считает агрегаты и разнообразие;
- делает **одношаговый** LLM-синтез слабых мест и правок промпта.

Цель этого этапа — приблизить систему к **исследовательски честной** оценке промптов:

- **A:** выводы мета-анализа опираются на **доказательства** из реальных ответов прогона, а не на общие рассуждения;
- **C:** пользователь видит **динамику** по серии прогонов (сначала по одному «объекту сравнения», затем — сравнение моделей);
- **D:** в UI и в сохранённых артефактах видно **откуда** взялся каждый вывод (result_id, цитата, при необходимости — шаг цепочки);
- **B (позже):** «экономный» пресет (меньше LLM-шагов / более лёгкая модель) после того, как надёжный контур зафиксирован.

Не цель v1 этого документа: полный programmatic checker UI (MVP-2), турнир Elo (MVP-3), публичные ссылки (MVP-4).

## 2. Зафиксированные приоритеты (из диалога)

| Приоритет | Содержание |
|-----------|------------|
| A + C | Достоверный мета-анализ **и** сквозной анализ многих прогонов |
| C3 | **Сначала C1**, затем **C2** (оба нужны, очередность жёсткая) |
| D | Доказательная трассировка выводов (обязательный контракт данных + UX) |
| B | Дешёвый режим — после базового контура |

## 3. Ключ сравнения (comparison key) и «линия времени»

Без явного ключа тренды по `run_id` бессмысленны: смешиваются разные задачи, промпты и модели.

### 3.1 Сущность lineage (логическая, не обязательно отдельная таблица в v1)

**Линия времени C1** (один промпт, одна задача, опционально одна модель для чистого тренда качества промпта):

- `prompt_fingerprint` — стабильный хэш текста промпта A (и B в pair), нормализованного по правилам (trim, возможно нормализация переводов строк);
- `task_fingerprint` — хэш `task_input` (и при наличии — `reference_answer`), те же правила;
- `library_anchor` — если прогон привязан к Library: `(prompt_a_library_id, prompt_a_library_version)`; иначе NULL;
- `target_model_id` — для C1 можно строить **две** линии: (a) фиксированная модель; (b) «все модели» с разбивкой по сериям (см. C2).

**Серия C2** (тот же промпт + та же задача, **разные** `target_model_id`):

- Тот же `(prompt_fingerprint, task_fingerprint)` [и тот же rubric preset / snapshot id, если сравнение честное];
- Группировка: по `target_model_id` → внутри — сортировка по `created_at`;
- В UI: мини-таблица или sparkline «p50 / diversity / доля провалов по критерию X» по моделям за выбранный период.

### 3.2 Правило честности

Прогоны с **разным** `task_fingerprint` или **разным** `rubric_snapshot` (существенное изменение критериев) **не склеиваются** в одну линию по умолчанию; UI показывает предупреждение «рубрика или задача изменились — сравнение ориентировочное» или требует явного «разрешить смешение» (опционально, бэклог).

## 4. Фазы поставки

### Фаза 1 — C1 + D (ядро)

1. **Хранение / вычисление** `prompt_fingerprint`, `task_fingerprint` при создании run (и при необходимости бэкфилл для старых прогонов — отдельная миграция или lazy compute при первом запросе).
2. **API / UI:** список прогонов в одной линии времени (фильтр: Library item, или «по хэшу», или «последние N»).
3. **Мета-анализ v2:** конвейер из нескольких шагов с **обязательным** `evidence[]` на каждый failure_mode / weak_spot (см. §5).
4. **UI доказательств:** раскрывающийся блок с цитатами и ссылками на `eval_results` (как в устной договорённости из brainstorming).

### Фаза 2 — C2

1. Тот же ключ `(prompt_fingerprint, task_fingerprint)` + согласованная рубрика → группировка по `target_model_id`.
2. Экран «Сравнение моделей» для выбранной пары/набора моделей: агрегаты side-by-side, те же failure modes где возможно.

### Фаза 3 — B (экономный пресет)

Пресет «Meta-lite»: один проход или укороченная цепочка, с явным бейджом «меньше гарантий, чем Meta+».

## 5. Мета-анализ: цепочка промптов и контракт D

### 5.1 Принципы

- **Детерминированный препроцессинг** где возможно: кластеризация выходов по эмбеддингам, пороги по оценкам судьи, подсчёт частот критериев.
- **LLM-шаги** с узкими ролями и фиксированным JSON-schema; запрет на «новые факты» без evidence.
- **Верификация в коде:** excerpt должен встречаться в `output_text` соответствующего `result_id` (после нормализации пробелов); иначе запись отбрасывается или помечается `confidence: low` и не показывается как жёсткий факт.

### 5.2 Пример цепочки (рекомендуемый минимум для «A»)

1. **cluster** — вход: список result_id + embeddings; выход: cluster_id → [result_id].
2. **hypothesize** — вход: кластеры + усечённые ответы + оценки судьи; выход: список гипотез `{ pattern, affected_cluster_ids, draft_evidence }`.
3. **verify** — вход: гипотезы; выход: только подтверждённые пункты с `evidence: [{ result_id, excerpt, criterion_key? }]` где excerpt проверен substring-match.
4. **synthesize_report** — вход: проверенные гипотезы + агрегаты; выход: summary, prompt_fixes, criteria_weak_spots — **каждый** пункт должен ссылаться на id гипотезы из шага 3 (трассировка).

Артефакты шагов сохраняются (JSON в БД или отдельная таблица `eval_meta_analysis_steps`) для аудита и отладки промптов мета-слоя.

### 5.3 Совместимость с текущим `synthesis_report_json`

- v1 поля остаются для обратной совместимости UI;
- новый формат либо версионируется (`schema_version: 2`), либо живёт в соседнем поле `meta_pipeline_json` — решение на этапе плана реализации (избегать дублирования в UX).

## 6. UX (кратко)

- **Страница прогона:** вкладка «Мета-анализ» с timeline шагов (опционально свёрнута по умолчанию).
- **Страница Library / Eval Studio:** «Тренд» для выбранного промпта: график p10/p50/p90, diversity, счётчики failure modes из последнего мета-отчёта.
- **C2:** переключатель «Группировать по модели» на том же экране тренда.

## 7. Риски и митигации

| Риск | Митигация |
|------|-----------|
| Дорогая цепочка LLM | Фаза 3 (Meta-lite); кэш по fingerprint + набору result_id |
| Галлюцинации на шаге hypothesize | Обязательный verify + substring check |
| Старые прогоны без fingerprint | Lazy backfill или пометка «тренд недоступен» |
| Пользователь меняет рубрику | Жёсткое разделение линий или предупреждение (§3.2) |
