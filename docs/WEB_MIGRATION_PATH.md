# Web Migration Path

## Цель

Переносить продукт в `FastAPI + React` только после того, как `Streamlit`-версия зафиксировала правильный product shape.

## Canonical flows, которые web-версия обязана сохранить

1. `Generate`
   - ввод задачи;
   - классификация;
   - выбор техник;
   - reasoning;
   - prompt output;
   - metrics.
2. `Iterate`
   - взять текущий промпт;
   - дать обратную связь;
   - получить новую версию.
3. `Questions`
   - получить уточняющие вопросы;
   - ответить;
   - форсировать финальный prompt generation.
4. `Library`
   - сохранить;
   - редактировать;
   - искать;
   - открывать обратно в инженере.
5. `Compare`
   - запускать только при честно отличающихся наборах техник.
6. `Metrics`
   - не терять локальную телеметрию продукта.

## Что уже есть в web-части

- `backend/`:
  - generate API
  - compare API
  - library CRUD
  - techniques API
- `frontend/`:
  - базовые страницы для Home / Compare / Library / Techniques / Settings

## Что расходится со Streamlit

1. `Q&A flow`
   - во frontend есть заглушка, а не завершенный пользовательский цикл.
2. `Compare`
   - в web-версии обе стороны сравнения пока легко сводятся к одинаковому автоподбору.
3. `Metrics`
   - в web-версии нет product telemetry UI.
4. `Version history`
   - Streamlit-сценарий богаче.

## Рекомендуемая последовательность миграции

### Шаг 1

Считать `core/`, `db/`, `services/` источником истины для доменной логики.

### Шаг 2

Перенести в web только такие сценарии:

- generate;
- iterate;
- library;
- techniques;
- Q&A.

### Шаг 3

Сравнить результаты `Streamlit` и `web` по одному и тому же evaluation set.

### Шаг 4

Только после этого переносить A/B compare как полноценную пользовательскую функцию.

## Decision gate

Не делать web-миграцию основным направлением, пока:

- Streamlit не стал сильным demo-продуктом;
- не собираются продуктовые метрики;
- не появились минимальные тесты на core.
