КРИТИЧНОДве независимые оси (уровень + модель) перегружают новичка
Пользователь видит: «Junior · Mid · Senior · Creative» И «DeepSeek V4 / Gemini / Mistral / Grok». Для него это 4×4 = 16 комбинаций без понимания разницы. Что выбрать? Документ сам признаёт проблему в разделе 10 — и предлагает решение (LevelBundle). Но это не реализовано.
→ Реализовать LevelBundle из раздела 10 документа. Каждый уровень = рекомендуемая модель + поведение. «Своя модель» — для продвинутых.
КРИТИЧНОtarget_model = «unknown» для Фото и Скилл — теряется контекст
При генерации промптов для изображений и скиллов бэкенд не знает под какую модель оптимизировать. Это значит model-specific hints не применяются. Для скиллов это особенно критично: промпт для Claude Agent ≠ промпт для GPT-4 Assistants API.
→ Для режима Скилл: добавить поле «Целевая среда» (Claude / GPT-4 / LangGraph / CrewAI / Общий) → передавать как target_model с маппингом на существующие карточки или как отдельное поле skill_target_env.
ВАЖНОTemperature cap 0.85 не реализован — документ предлагает, код не делает
Creative использует T=0.92 что сам документ называет проблемой («поломки формата»). Предложение ввести потолок есть в документе но не реализовано. Creative при T=0.92 регулярно нарушает [PROMPT] блок.
→ Одна строка в applyExpertPreset: temperature: Math.min(preset.temperature, 0.85). Или серверный cap в generate.py. Для Creative добавить tooltip «Высокое разнообразие — иногда требует повторной генерации».

Тренды апрель 2026 — что происходит в AI с выбором моделей

ТРЕНД 1Model routing по задаче — автоматический выбор модели
OpenRouter, LangChain, Anthropic — все двигаются к auto-routing: задача отправляется к наиболее подходящей модели автоматически. GPT-4o для reasoning, Claude Haiku для форматирования, Gemini Flash для скорости. Пользователь не выбирает модель — он выбирает цель.
→ Для MetaPrompt: уровень «Авто» где система сама выбирает модель по типу задачи (classification → дешёвая, generation → качественная, скилл → reasoning). Пользователь видит только результат и цену запроса.
ТРЕНД 2Reasoning models (o3, R1) требуют другой логики уровней
o3, DeepSeek R1, Claude Extended Thinking — у них встроенный CoT. Уровень «Senior» с manual техниками включая chain_of_thought бессмысленен для этих моделей — они делают это сами, и добавление CoT в промпт только мешает. Текущая система уровней это не учитывает вообще.
→ При выборе reasoning-модели (o3, R1, Sonnet Extended) — автоматически переключать технический пресет: убирать CoT из manual_techs, снижать temperature до 0.3–0.5, отключать questions_contract. Добавить флаг is_reasoning_model в model catalog.
ТРЕНД 3Cost visibility — пользователи хотят знать цену до клика
Vercel AI SDK, OpenRouter UI, Anthropic Console — все показывают предполагаемую стоимость запроса до отправки. Это критично при выборе между «Creative + Фото + Deep mode» (3 LLM вызова) и «Junior + Текст» (1 вызов).
→ Рядом с кнопкой «Создать» показывать: «~$0.0003 · 1 вызов» или «~$0.002 · 3 вызова». Данные из LevelBundle + текущей модели. Это также объясняет почему Senior иногда быстрее Junior несмотря на «сложнее».

Разбор новых скриншотов

Скрины 1–9 — детальный разбор каждого

Студия — главный экран (скрины 1–2)

ОТЛИЧНОDiff «Что изменилось V1→V2» + спарклайн версий — это реализовано!
«Убрано 1 строка · Добавлено 20 строк · Полнота: 25%→65%» — именно это я предлагал в прошлом ревью. Версии v3 80% · v2 65% · v1 25% с маленьким спарклайном графика — killer UX который нигде у конкурентов нет.
→ Это самая сильная уникальная фича продукта. Сделать версии кликабельными для отката — тогда это станет настоящим undo-history для промптов.
ХОРОШО«Совет 1 / Совет 2 / Совет 3» как chips + «Применить всё»
Предложенная система chips-советов реализована. Кнопка «Применить всё» вверху — правильно.
→ Следующий уровень: советы должны быть специфичными для конкретного промпта («Добавь пример для JSON-структуры»), а не общими («Few-Shot»). Генерировать их как часть основного ответа.
ВАЖНО«УРОВЕНЬ Junior · Mid · Senior · Creative» — строчка под вкладками
4 кнопки уровня + надпись «УРОВЕНЬ» занимают целую строку. Это 20px высоты для функции которую большинство пользователей не меняет. При открытии студии первое что видишь — три строки навигации: Текст/Фото/Скилл + Уровень + метаданные токенов.
→ Уровень убрать из основного UI. Разместить в тулбаре справа как маленький badge-дропдаун: «Mid ▾». Кликнул — раскрылся список. Экономит 20px и убирает визуальный шум.
ВАЖНО«≈66 tok исходный текст» — непонятный лейбл
Строка под кнопками уровней показывает «≈66 tok исходный текст». Что это? Размер входящего текста задачи? Размер предыдущего промпта? Для пользователя неочевидно.
→ Переименовать в понятный контекст: «Задача: ~66 токенов» или убрать совсем — пользователю это не нужно на глаза.

A/B Сравнение (скрин 3)

ВАЖНООбъяснение «Авто/Вручную» — текстовая стена
«Авто — техники подбираются по типу задачи. Вручную — только отмеченные ниже техники попадут в промпт. После генерации в каждом столбце появится текст промпта и метрики — так видна разница А и В.» — длинный абзац который никто не читает.
→ Заменить на две строки tooltip при hover на «Авто» и «Вручную». Основной экран — чистым.

Библиотека (скрин 4)

СИЛЬНОНазвания промптов теперь человеческие — большой прогресс
«Неформальная заметка с живым потоком мыслей», «Техники составления промптов для исследователей ИИ» — реальные человеческие названия. Это был главный запрос из всех ревью.
→ Отлично. Теперь добавить сортировку «По последнему использованию» — самые актуальные наверху.
ВАЖНОКарточки: «Открыть» + «В чат» + «Копировать» + «Ещё» — неравномерная иерархия
«В чат ▾» — дропдаун со стрелкой рядом с «Открыть» — создаёт путаницу. «Ещё» в правом углу прячет важные действия. Пользователь не знает куда нажать.
→ Одна primary кнопка «Открыть». Остальное — hover-reveal или «...» меню. Сейчас 4 контрола первого уровня на карточку — много.

Скиллы (скрины 5, 7, 8)

БОЛЬШОЙ ПРОГРЕСССкилл теперь генерирует структуру ROLE/SCOPE/RULES/PROCEDURE
Скрин 8: сгенерированный скилл содержит секции ROLE, SCOPE (входящие/исходящие/не входит), RULES (4 правила), PROCEDURE (3 шага). Это именно та структура которую я предлагал в предыдущем анализе. Плюс «Песочница» и «Тест-кейсы (3)» — это фактически Eval Studio!
→ Это одна из сильнейших реализаций в продукте. Скилл с тест-кейсами — дифференциатор рынка. Продвигать это как главную фичу.
ВАЖНО«Скиллы хранятся локально в браузере» — огромный UX-риск
Скрины 5 и 7: текст «Скиллы хранятся локально в браузере» встречается несколько раз. Это значит очистка кеша = потеря всех скиллов. Для пользователя который потратил час на создание скиллов — это катастрофа.
→ СРОЧНО: перенести скиллы на сервер (как промпты). Или хотя бы: кнопка «Экспорт всех скиллов» + предупреждение при выходе если есть несохранённые. Локальное хранение — антипаттерн для ценных данных.
ВАЖНО«Базовый текстовый навык для Claude» — «давай сделаем скилл»
Скрин 7: скилл называется «Базовый текстовый навык для Claude» но описание — «давай сделаем скилл». Это первая строка из чата задачи, не название скилла. Проблема auto-naming не полностью решена для скиллов.
→ При сохранении скилла — AI-генерация названия по первой секции ROLE. «Ты — текстовый ассистент» → «Текстовый ассистент без внешних ресурсов». Автоматически.

Сообщество (скрин 6)

ХОРОШОИзображение в карточке сообщества — визуальный preview для фото-промптов
Карточка «Надо нарисовать черепашки ниндзя которые грабят м...» содержит AI-сгенерированное изображение. Это правильно — фото-промпт должен показывать результат, а не текст.
→ Расширить: для текстовых промптов — показывать краткое описание задачи. Для скиллов — иконку типа скилла. Разные визуальные паттерны для разных типов контента.

Дропдаун моделей (скрин 9)

ВАЖНО«google/gemini-flash-1.5» в дропдауне — техническое название
Список моделей: «DeepSeek V4», «google/gemini-flash-1.5», «Mistral Nemo», «Grok 4.1 Fast». Второй пункт — сырой OpenRouter ID без нормализации. Несоответствие стиля.
→ Применить shortGenerationModelLabel из кодовой базы ко всем моделям в списке. «google/gemini-flash-1.5» → «Gemini Flash 1.5». Добавить иконки провайдеров.

Продуктовый брейнсторм

Критический взгляд на продукт — что есть, чего не хватает, куда двигаться

Что продукт реально умеет (сильные стороны)

✓ Генерация структурированных промптов из описания задачи — уникально на рынке
✓ Версионирование v1→v7 со score и diff — нигде у конкурентов нет
✓ Три режима: Текст / Фото / Скилл — охват разных use cases
✓ Скилл с ROLE/SCOPE/RULES + тест-кейсами — фактически Eval lite уже работает
✓ Сообщество с публикацией — growth механика
✓ A/B сравнение техник — образовательная ценность

Что мешает росту — честный разбор

ГЛАВНАЯ ПРОБЛЕМАПродукт не объясняет зачем он нужен при первом открытии
Новый пользователь видит: чат + поле задачи + переключатели. Он не понимает: «Чем это отличается от ChatGPT? Зачем мне промпт-оптимизатор если я могу просто написать в ChatGPT?» Нет onboarding flow, нет «до/после» демо, нет объяснения ценности.
→ Первый запуск: интерактивное демо. Показать один промпт «до» и результат «после» — прямо в интерфейсе без регистрации. «Попробуй сам → введи задачу».
ВАЖНОНет retention механики — пользователь сделал промпт и ушёл
Библиотека растёт но пользователи не возвращаются к промптам. Нет «коллекций», нет проектов, нет уведомлений «ваш промпт использован 5 раз в сообществе». Скиллы в localStorage — антипаттерн для retention.
→ «Проекты» как связка промптов + скиллов + пространств для конкретного usecase. «Мой проект: контент для YouTube» = набор промптов + скилл-редактор + пространство с брендом.
ВАЖНОПозиционирование размыто: для кого это?
Сейчас продукт пытается быть и для no-code (Junior, простые режимы) и для prompt engineers (Senior, manual техники, A/B). Это разные аудитории с разными ценностями. No-code хочет «просто работает», prompt engineer хочет «полный контроль».
→ Выбрать основную ЦА для беты: prompt engineers и разработчики AI-продуктов. Они понимают ценность, готовы платить, помогут с feedback. No-code — вторая волна после product-market fit.
BIG IDEA«Playground» — тестирование промпта прямо в продукте
«Песочница» уже есть в скилл-режиме! Это зародыш Eval Studio. Расширить: для любого промпта — playground где можно ввести тестовый input и увидеть output прямо в MetaPrompt. Не нужно копировать в ChatGPT.
→ POST /playground: берёт промпт + user input → прогоняет через выбранную модель → показывает output. Score считается автоматически. Это замыкает loop: создал → протестировал → улучшил → сохранил.
BIG IDEA«Prompt Templates» — sharable templates для типовых задач
Большинство пользователей делают одинаковые вещи: summarize, classify, extract, generate. Если MetaPrompt предложит стартовые шаблоны под каждый usecase (с уже выбранными техниками и уровнем) — friction при первом использовании упадёт кратно.
→ «Начать с шаблона»: 10–15 готовых сценариев (Анализ документа, Генерация контента, Код-ревью, Парсинг данных...). Кликаешь → заполнена задача + выбран режим + применены оптимальные техники.
МЕТА-НАБЛЮДЕНИЕПродукт стал сложным быстрее чем стал понятным
За несколько месяцев добавились: 4 уровня + 3 режима + A/B + скиллы + сообщество + пресеты + пространства + модели. Каждая фича правильная — но вместе они создают cognitive overhead. Новый пользователь тонет.
→ Принцип: сначала один путь к успеху. Главный экран = одно действие: «Опиши что нужно → получи промпт». Все остальные настройки — progressive disclosure. Сложность открывается по мере использования.

Анкета после теста (обязательные вопросы)

1. «Объясните продукт другу за одно предложение» — диагностика понимания value prop
2. «Что было непонятно без объяснений?» — список friction points
3. «Что вам понравилось больше всего?» — подтверждение differentiators
4. «За что вы заплатили бы?» — сигнал willingness to pay
5. «Что вам НЕ хватило для полноценной работы?» — следующий roadmap
6. NPS 0–10: «Порекомендуете ли коллеге?» — стандартная метрика
7. «Какой вы сделали промпт? Использовали ли его в реальной работе?» — реальное применение
