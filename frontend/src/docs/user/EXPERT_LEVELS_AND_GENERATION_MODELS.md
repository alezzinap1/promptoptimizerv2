# Уровни эксперта и модели генерации: полное описание

Документ описывает **текущую реализацию** в кодовой базе MetaPrompt (студия на **Home**), **техническую цепочку** от UI до бэкенда, **все дополнительные вызовы LLM** в одном пользовательском «запуске», а также **продуктовые идеи**: объединить выбор уровня с подбором моделей, сохранив ручной выбор модели. Отдельно — обоснование **потолка температуры 0.85** для снижения сбоев формата.

---

## 1. Термины

| Термин | Смысл |
|--------|--------|
| **Уровень эксперта** (`ExpertLevel`) | Один из пресетов: `junior` \| `mid` \| `senior` \| `creative`. Не отдельный API-режим: это набор согласованных настроек студии. |
| **Режим студии** (`PromptStudioMode`) | Вкладка: `text` \| `image` \| `skill`. Один и тот же уровень даёт **разные** поля пресета для разных вкладок (см. `getExpertLevelPreset`). |
| **Модель генерации** (`gen_model`) | Модель OpenRouter, которой выполняется **основной** (и часть вспомогательных) вызовов при `/generate`. На фронте — выпадающий список в композере чата. |
| **Целевая модель** (`target_model`) | Логическая «модель, под которую пишется промпт»: влияет на **сборку system/user контекста** (инструкции под семейство модели). Для режимов **Фото** и **Скилл** в запрос уходит `unknown`. |
| **`expert_level` в теле запроса** | Строка, дублирующая выбранный уровень. На бэкенде используется **дополнительно** к уже присланным флагам — в основном для тонкой подстройки лимита уточняющих вопросов. |

---

## 2. Где пользователь выбирает уровень

### 2.1. UI

- Страница: **`Home`** (`frontend/src/pages/Home.tsx`).
- Место: под заголовком «Задача», рядом с переключателем **Текст / Фото / Скилл**.
- Контрол: ряд кнопок **Junior · Mid · Senior · Creative** (`expertLevelRow`).
- Подсказки: `EXPERT_LEVEL_HINTS` из `frontend/src/lib/expertLevelPresets.ts` (атрибут `title` у кнопок).

### 2.2. Обработчик смены уровня

При клике вызывается `handleExpertLevelChange`:

1. Сохраняется `expertLevel` в состоянии React.
2. Вызывается `applyExpertPreset(level, promptType)` — см. раздел 3.
3. В `studioModesRef.current[promptType]` обновляется поле `expertLevel` (чтобы при переключении вкладок уровень для текущего режима был согласован).

Важно: **модель генерации** (`genModel`) при смене уровня **не меняется** — уровень и модель сейчас **независимы**.

## 3. Как настраивается уровень (пресет)

Источник истины: **`getExpertLevelPreset(level, mode)`** — файл `frontend/src/lib/expertLevelPresets.ts`.

Функция **`applyExpertPreset`** в `Home.tsx` применяет результат к состоянию:

- `questionsMode` — вкл/выкл сценария уточняющих вопросов (на первом проходе, пока нет ответов).
- `techniqueMode` — `auto` или `manual`.
- `manualTechs` — список id техник из YAML-базы (для `manual`).
- `temperature`, `topP` — сэмплинг основной генерации.
- `imageDeepMode` — только если поле задано в пресете (переключатель «Анализ сцены» на вкладке Фото может быть выставлен пресетом).

### 3.1. Таблица пресетов (по коду)

| Уровень | Вкладка | Вопросы | Техники | `manualTechs` | T | top_p | Анализ сцены |
|---------|---------|---------|---------|---------------|---|-------|--------------|
| Junior | любая | вкл | auto | `[]` | 0.45 | 0.95 | выкл |
| Mid | любая | вкл | auto | `[]` | 0.7 | 1 | выкл |
| Senior | любая | выкл | manual | `SENIOR_MANUAL_TECHNIQUE_IDS` (5 id) | 0.85 | 1 | **вкл только image** |
| Creative | text | вкл | manual | первые **4** из того же набора | 0.92 | 1 | выкл |
| Creative | image | вкл | auto | `[]` | 0.92 | 1 | **вкл** |
| Creative | skill | выкл | auto | `[]` | 0.92 | 1 | н/д |

Константа **`SENIOR_MANUAL_TECHNIQUE_IDS`**: `chain_of_thought`, `role_prompting`, `self_consistency`, `constraints_prompting`, `meta_prompting`.

### 3.2. Поведение при смене вкладки (Текст ↔ Фото ↔ Скилл)

При переключении режима вызывается `hydrateFromSnapshot`, который:

1. Восстанавливает сохранённый для этой вкладки `expertLevel` (или `mid` по умолчанию).
2. **Снова** вызывает `applyExpertPreset(lvl, mode)` для **новой** вкладки.

Итог: если пользователь выбрал **Creative** на тексте, перешёл на **Фото** — для фото подтянется **тот же** сохранённый уровень вкладки «image» из снимка (изначально `mid` в `createEmptyStudioSnapshot`), а не «текущая кнопка с текстовой вкладки». У каждой вкладки свой снимок и свой запомненный `expertLevel`.

### 3.3. Персистентность

- **`expertLevel`** хранится в **`AgentStudioSnapshot`** (`frontend/src/lib/agentStudioModes.ts`) **отдельно для text / image / skill**.
- Черновик в `localStorage`: **`saveAgentDraftV2` / `loadAgentDraftV2`** (`frontend/src/lib/agentDraft.ts`). Старые черновики без поля получают `expertLevel: 'mid'`.

**Модель генерации** в снимок студии **не входит** — она глобальна для сессии страницы и берётся из настроек при загрузке.

### 3.4. Рекомендуемые модели по уровню (`EXPERT_DEFAULT_GEN_MODEL`)

Константа в **`frontend/src/lib/expertLevelPresets.ts`**: при смене уровня UI может подставить соответствующий OpenRouter id (если пользователь не закрепил другую модель вручную). Идентификаторы сверены с каталогом [OpenRouter Models](https://openrouter.ai/models) (актуальность — периодически перепроверять).

| Уровень | OpenRouter id | Заметка |
|---------|-----------------|---------|
| Junior | `deepseek/deepseek-v4-flash` | Низкая цена, сильный русский |
| Mid | `google/gemini-2.5-flash-lite` | Flash Lite: скорость и экономия |
| Senior | `google/gemini-2.5-flash` | Flash: выше потолок качества |
| Creative | `openai/gpt-4o-mini` | Баланс креатив/стоимость |

## 4. Что уходит на сервер при генерации

Тип запроса: **`GenerateRequest`** (см. `frontend/src/api/client.ts` и `backend/api/generate.py`).

Поля, **прямо связанные с уровнем** (выставляются из состояния UI после пресета):

| Поле | Смысл |
|------|--------|
| `expert_level` | Строка: `junior` \| `mid` \| `senior` \| `creative`. |
| `questions_mode` | На фронте: `questionsMode && !questionAnswers?.length` — на **первом** проходе без ответов true, после ответов на вопросы false. |
| `technique_mode` | `auto` \| `manual`. |
| `manual_techs` | Список id; при `auto` уходит `[]`. |
| `temperature`, `top_p` | Числа с панели (заданы пресетом, пользователь может менять в «Дополнительно»). |
| `image_deep_mode` | Только для `prompt_type === 'image'`. |

Поля **моделей**:

| Поле | Откуда на фронте |
|------|------------------|
| `gen_model` | `genModel` — выбранный пункт в списке «Модель генерации». |
| `target_model` | `effectiveTargetModel`: для **текста** = `targetModel`; для **image/skill** принудительно **`unknown`**. |

## 5. Бэкенд: как используется `expert_level`

Файл **`backend/api/generate.py`**.

### 5.1. Политика вопросов

После `get_questions_policy(...)` вызывается **`_apply_expert_level_questions_policy`**, которая меняет только **`max_questions`**:

- **junior**: `max(текущий, 3)`, не выше 5.
- **senior**: не выше 2 (минимум 1).
- **creative**: `min(5, текущий + 1)`.
- **mid** и неизвестное значение: без изменений от этого хелпера.

Важно: при **`questions_mode: false`** (типично **Senior**) логика «первичной генерации с неотвеченными вопросами» не активируется — см. `_is_primary_generation_with_unanswered_questions`.

### 5.2. Остальное поведение уровня

Задаётся **фронтом**: техники, температура, флаги вопросов, глубокий режим картинки. Бэкенд **не** содержит отдельной ветки «если junior — другая модель».

## 6. Модель генерации (`gen_model`): полный путь

### 6.1. Откуда список в UI

1. При монтировании `Home` запрашиваются **`api.getSettings()`** и **`api.getModels()`**.
2. **`preferred_generation_models`** из настроек пользователя кладётся в `generationOptions`.
3. Подписи к id подгружаются из каталога OpenRouter (`modelsRes.data` → `modelLabels`).
4. Селект строится как **`genModelSelectOptions`**: только id из `generationOptions`, label — укороченное имя (`shortGenerationModelLabel`).

Если локальное состояние `genModel` пустое, подставляется **`settings.preferred_generation_models[0]`**.

### 6.2. Где ещё задаётся список

- API **`PATCH /settings`** (`backend/api/settings.py`): поле `preferred_generation_models`.
- Хранение: payload пользователя через `user_preferences` (см. `get_user_preferences_payload`).

### 6.3. Что делает бэкенд с `gen_model`

- Резолв в полный OpenRouter id: **`PROVIDER_MODELS`** или строка уже с `/` — см. `_get_openrouter_model_id` в `generate.py`.
- Словарь коротких имён: **`services/llm_client.py`** (`PROVIDER_MODELS`, `DEFAULT_PROVIDER = "deepseek"`).
- **Основная генерация**: `llm.stream(system_prompt, user_content, req.gen_model, req.temperature, ...)`.
- **Пробный режим** (без своего ключа): проверка **стоимости completion** модели; дорогие модели блокируются.

### 6.4. Связь с качеством ответа

В UI есть текстовые подсказки по **issue** (например, нарушение формата `[PROMPT]` / `[QUESTIONS]`). Это **не привязано** к уровню: слабая инструкция следования формату — функция **выбранной модели генерации** и температуры.

## 7. Целевая модель (`target_model`)

### 7.1. Назначение

Это **не** модель, которая исполняет запрос к OpenRouter в `/generate`. Это **метаданные задачи**: «промпт готовится для GPT / Claude / … / любой».

- Влияет на сборку контекста в **`build_preview_payload` / `ContextBuilder`** (через `target_model` в запросе).
- Для **Фото** и **Скилл** в API уходит **`unknown`**, потому что выбор целевой LLM в композере заменён на пресеты/движок изображения.

### 7.2. Откуда список в UI (только текст)

- **`preferred_target_models`** из `getSettings()`.
- Опция **`unknown`**: «Любая модель».
- При смене workspace: если в **`workspace.config.preferred_target_model`** задано не `unknown`, **`targetModel`** перезаписывается (effect в `Home.tsx`).

### 7.3. Справочник имён на бэкенде

В **`services/llm_client.py`** есть **`TARGET_MODELS`** — человекочитаемые подписи для известных ключей; фронт в основном опирается на каталог OpenRouter и пользовательский список.

## 8. Все дополнительные вызовы LLM (кроме основного стрима)

Один пользовательский «Запустить генерацию» → один HTTP **`POST /generate`**. Внутри может быть **несколько** обращений к моделям.

### 8.1. Классификация задачи (опционально)

- Условие: в настройках **`task_classification_mode === 'llm'`** и нет подходящего кеша классификации из сессии.
- Вызов: **`classify_task_with_llm`** — **отдельный** LLM-вызов **до** основной генерации.
- **Не зависит от уровня эксперта** напрямую; зависит от пользовательских настроек.

### 8.2. Анализ сцены (только изображения)

- Условие: `prompt_type == "image"` и **`image_deep_mode`**.
- Вызов: **`llm.generate(SCENE_ANALYSIS_SYSTEM, ...)`** с перебором провайдеров при ошибках.
- **Senior / Creative** на вкладке Фото выставляют пресетом **`imageDeepMode: true`** → этот шаг **включён по умолчанию** для этих уровней на фото.
- **Junior / Mid** пресетом держат **`imageDeepMode: false`** (пользователь может вручную включить «Анализ сцены» в тулбаре).

### 8.3. Основная генерация

- Всегда: **`llm.stream(...)`** с **`gen_model`** и **`temperature` / `top_p`** из запроса.

### 8.4. «Контракт» на уточняющие вопросы

- Условие: включён сценарий первичных вопросов, политика не `skip`, модель вернула промпт без корректного блока вопросов, сработал **`_should_enforce_questions_contract`** (в т.ч. большой `context_gap` или режим `required`).
- Вызов: дополнительный **`llm.generate`** с шаблоном `QUESTIONS_CONTRACT_TEMPLATE` и **`temperature=0.35`**.
- Чаще актуально для уровней с **`questions_mode: true`** на первом проходе. У **Senior** обычно `questions_mode: false` → эта ветка **не используется** для первого прохода.

### 8.5. Второй HTTP-запрос от пользователя

Если в ответе только вопросы, пользователь заполняет ответы и снова жмёт отправку — это **второй** `POST /generate` (новый стрим; `questions_mode` уже false, т.к. есть `question_answers`).

**Вывод по «где больше запросов»:**

- **Больше всего LLM-вызовов в одном `/generate`** у **Фото + анализ сцены**: 1 (сцена) + 1 (стрим) [+ ретраи сцены] [+ контракт вопросов, если включены вопросы].
- **Больше раундов с точки зрения пользователя** (несколько нажатий) — когда включены **уточняющие вопросы** (Junior, Mid, Creative для text/image и т.д.).

## 9. Температура, формат ответа и потолок 0.85

### 9.1. Текущие значения пресетов

- Junior: **0.45**
- Mid: **0.7**
- Senior: **0.85**
- Creative: **0.92**

### 9.2. Почему высокая температура ломает UX

Пайплайн ожидает **парсибельный** ответ с блоками вроде **`[PROMPT]`** и **`[QUESTIONS]`** (см. разбор на бэкенде после стрима). При завышенной температуре модели чаще:

- нарушают шаблон;
- смешивают вопросы и промпт;
- дают пустые или «заглушечные» варианты ответов.

В интерфейсе это превращается в баннеры ошибок и необходимость повторять генерацию.

### 9.3. Предложение: потолок **0.85**

**Идея:** при любой комбинации уровня и ручных правок не поднимать **`temperature`** выше **0.85** для основной генерации в студии (или показывать предупреждение и мягко капить значение).

- **Senior** уже на0.85 — выравнивание **Creative** с 0.92 → 0.85 уменьшит хвост «креативных» сбоев формата.
- **0.85** — компромисс: ещё заметное разнообразие формулировок при лучшей устойчивости инструкций, чем у 0.92+.

Технически потолок можно ввести:

- в **`applyExpertPreset`** / общей функции «применить сэмплинг»;
- или в **одном месте** перед `api.generate` (clamp);
- опционально — дублировать на бэкенде как защиту от произвольных клиентов.

Отдельно: вспомогательные вызовы (анализ сцены, контракт вопросов) уже используют **низкую** температуру (например 0.35) — их не нужно поднимать.

## 10. Идеи: объединить «уровень» и «модель» в один выбор

Сейчас пользователь выбирает **две независимые оси**: уровень (пресет поведения) и модель генерации (качество/стоимость/стиль рассуждений). Это гибко, но перегружает новичка.

### 10.1. Целевой UX

1. **Главный переключатель** — условный **«Профиль»** или расширенный **«Уровень»**, за которым скрыты:
   - рекомендуемая **модель генерации** (или небольшой набор «рекомендовано / быстрее / точнее формат»);
   - логика **вопросов / техник / анализа сцены / температуры** (как сейчас в пресете);
   - явная метка **«ожидаемое число LLM-вызовов»** (например: «обычно 1; с уточнениями до 2; фото + анализ сцены минимум 2»).

2. **Расширенный режим** — чекбокс **«Своя модель»**:
   - открывает текущий селект `gen_model` без ограничений списка из профиля;
   - при выключении — возврат к модели, привязанной к профилю.

3. **Целевая модель** (`target_model`) остаётся отдельным контролом для **текста** (как сейчас), либо тоже включается в профиль для текста («пишем под Claude», «пишем под GPT»).

### 10.2. Структура данных (концепт)

Ввести конфигурацию уровня расширенного типа, например:

```ts
type LevelBundle = {
  id: ExpertLevel // или новый enum «профиль»
  behavior: ExpertLevelPreset // как сейчас
  generationModel: {
    defaultGenModelId: string
    /** Дешёвый fallback, если основная недоступна в пробном режиме */
    trialFallbackId?: string
    /** Необязательный список «разрешённых» для этого уровня */
    allowedGenModels?: string[]
  }
  notes?: { expectedLlmCalls: string }
}
```

Профили можно хранить:

- на фронте (как сейчас пресеты) + опционально **синхронизировать** с настройками пользователя;
- или на бэкенде в `user_preferences` JSON, чтобы править без релиза фронта.

### 10.3. Связь с дополнительными запросами

В UI профиля полезно показывать **не** только модель, но и **сценарий**:

| Элемент сценария | Зависит от |
|------------------|------------|
| Уточняющие вопросы | `questions_mode`, политика gap, `expert_level` → `max_questions` |
| Второй проход после ответов | действия пользователя |
| Анализ сцены | `image_deep_mode` (пресет уровня + ручной тумблер) |
| Контракт вопросов | эвристики после первого ответа |
| LLM-классификатор | настройки пользователя, не уровень |

Так пользователь понимает: **«Creative + Фото»** ≠ **«Mid + Фото»** не только по температуре, но и по **минимальному числу вызовов**.

### 10.4. Сохранение ручной модели

Рекомендации:

- Флаг **`useCustomGenModel: boolean`** в локальном состоянии или в черновике.
- При **включённом** флаге не перезаписывать `genModel` при смене уровня; при **выключенном** — при смене уровня подставлять модель из бандла.
- В теле запроса по-прежнему уходит явный **`gen_model`** — бэкенду не обязательно знать «профиль», если не нужна аналитика.

### 10.5. Риски

- **Пробный режим**: бандл должен указывать модели в пределах **`TRIAL_MAX_COMPLETION_PER_M`** или автоматически падать на fallback.
- **Каталог OpenRouter** меняется: id моделей лучше хранить **полными** (`provider/model`), короткие ключи — только как алиасы.
- Пользователи, привыкшие к **Creative @ 0.92**, при капе **0.85** могут заметить «меньше разнообразия» — это стоит описать в подсказке профиля.

## 12. Краткое резюме

- **Уровень** — это **пресет** вопросов, техник, сэмплинга и (для фото) анализа сцены; выбирается на **Home**, хранится **по вкладке** в снимке студии.
- **`expert_level`** на сервере в основном **подкручивает лимит вопросов**; всё остальное уже в явных полях запроса.
- **Модель генерации** и **целевая модель** — разные роли; вторая для image/skill в запросе **`unknown`**.
- **Дополнительные LLM-вызовы**: классификатор (настройка), анализ сцены (флаг), контракт вопросов (эвристика), плюс **повторный** `/generate` после ответов на вопросы.
- **Объединение уровня с моделью** — продуктово разумно как **бандл + флаг «своя модель»**; **потолок температуры 0.85** — инженерная мера против поломок формата, особенно если опустить Creative с 0.92.
